# 入职 90 天生存指南（#22）

> 触发时机：用户接受 Offer 后，career-coach 自动生成。

## 核心原则

入职前 90 天是最危险的时期。统计显示 30% 的新员工在试用期离职或被动离职。本指南帮你安全渡过。

## 第 1-30 天：融入期

### 目标：理解环境、建立信任、不犯错

**第一周：**
- 搞清楚决策链：谁是你的直接汇报对象？谁影响你的绩效评估？谁控制资源？
- 记录所有 onboarding 文档的遗漏和矛盾（这是你的第一个贡献机会）
- 问"为什么"而非建议"应该怎样"（前两周你的任何建议都会被看作天真）

**第二-四周：**
- 找一个盟友（非正式导师）：组织里的老员工，不是你的 manager
- 完成第一个小 win（修一个 bug、改进一个文档）——必须在一月结束前有可见贡献
- 摸清团队的政治地图：谁和谁有矛盾、谁的 tech opinion 最有分量

### 关键动作
- [ ] 首次 1:1 问 manager：「你最希望我在 30/60/90 天内做成什么？」
- [ ] 了解绩效评估流程和标准（KPI/OKR）
- [ ] 阅读团队最近 3 个月的 design doc 和 postmortem

## 第 31-60 天：影响力建设期

### 目标：从新人变成靠谱的团队成员

- 找到一个你比团队其他人更擅长的小领域，成为 go-to person
- 参与 code review，建立技术标准意识（不是挑刺，是提升质量）
- 开始写 design doc（即使是小功能的）
- 建立跨团队联系：找到与你的工作有接口的其他团队

### 关键动作
- [ ] 独立交付一个中等复杂度的功能
- [ ] 在团队会议上做一次技术分享
- [ ] 开始记录你的贡献（为试用期 review 准备）

## 第 61-90 天：独立贡献期

### 目标：证明转正价值

- 承担一个有可见业务影响的 initiative
- mentor 新人或实习生（证明你的成长潜力）
- 开始建立外部可见度（技术博客、开源贡献）

### 试用期 review 准备
- 量化你的 90 天贡献：修了多少 bug、交付了多少 feature、提高了什么指标
- 收集 peer feedback（至少 3 人）
- 与 manager 对齐：接下来 6 个月的成长计划

## 红色警告（危险信号）

| 信号 | 应对 |
|------|------|
| Manager 连续 2 次 1:1 取消或敷衍 | 这是最大危险信号，开始更新简历 |
| 团队频繁有成员离职 | 了解离职原因，判断是否系统性 |
| 你的工作内容与 JD 严重不符 | 第 30 天前与 manager 沟通纠正 |
| 被排除在关键会议之外 | 主动申请参加，问「我需要做什么才能参与？」 |
| 试用期 review 没有任何反馈 | 主动发起，不要等 |

## 特别提醒

- 这里不是「免费试用」，你有价值才被录用
- 技术能力只占成功因素的 50%，组织理解能力占另外 50%
- 保持与前同事和猎头的弱联系（备用网络）
